\section{Data Collection and Methodology} %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%


The data set I analyzed was collected during winter semester 2023/2024 in the module \href{https://modhb.rptu.de/mhb/modules/INF-02-01-M-2/}{Concepts of Programming}.
The data was anonymized and forwarded to me by my supervisor. There were \(220\) students in total submitting their files individually.
During this semester the students had to complete \(12\) sheets with \(16\) tasks that are relevant.
\textit{Relevant} in this case means that these were coding tasks with unit tests to automatically detect the correctness of the submission.
The students were provided with a template project containing the following:

\begin{itemize}[nosep]
    \item A file called \texttt{Mini.fs} -- a small library of useful methods and basic types that are used throughout all of the assignments.
        For instance \texttt{Mini.fs} simplifies the user input and output to shift the focus away from all possible edge cases of I/O this early on.
        Mostly used is the type \fsharp|Nat|, describing a natural number including zero (\(\mathbb{N}_0\)),
        written as a non-negative integer with postfix `\texttt{N}': \fsharp|0N|, \fsharp|1N|, \(\dotsc\), \fsharp|1234N|, etc.
    \item Further assignment-specific types (e.g. type definition of a linked list and the individual nodes)
    \item All relevant methods with their method signature and a placeholder\\
        \fsharp|failwith "TODO"| for the body
\end{itemize}

\noindent The unit tests normally consist of:
\begin{itemize}[nosep]
    \item Example tests that verify edge cases and specific method calls given in the problem definition.
        If the goal is to implement a method, the problem definition lists multiple method calls that emphasize the functioning of said method.
    \item Random tests that verify an implementation against random input.
\end{itemize}

\noindent They did not have access to these tests, but they were able to execute them on the same platform where they submit their solutions (\href{https://exclaim.cs.rptu.de}{ExClaim}).
They do not have to execute any commands or change working directories in a terminal, but can rather verify their solutions individually by a simple button press.
Through this in particular I had access to all submissions for a single assignment -- in other words the history -- to analyze the progression.\\[5pt]
To illustrate, consider a method that returns the minimum of \(3\) natural numbers. The template for this method might look as follows:

\begin{fsharpcodeshort}{Template}{}
let minimumOfThreeNumbers (a: Nat) (b: Nat) (c: Nat): Nat =
    failwith "TODO"
\end{fsharpcodeshort}

\noindent The method is implemented with the placeholder \fsharp|failwith "TODO"| to ensure compilation even if not all methods are implemented so that they are still able to test their solution at any point.
The students are supposed to replace this \fsharp|failwith "TODO"| with their own implementation of that method. The problem definition then might list:
\begin{center}\begin{multicols}{2}
    \noindent
    \fsharp|minimumOfThreeNumbers 1N 2N 3N|
    \fsharp|minimumOfThreeNumbers 2N 3N 1N|
    \fsharp|minimumOfThreeNumbers 0N 0N 0N|
    \fsharp|minimumOfThreeNumbers 2N 2N 3N|
\end{multicols}\end{center}
\noindent to illustrate that neither the order of the input nor duplicates influence the end result. So exactly these method calls would be used for the example tests.

\begin{fsharpcodeshort}{Sample Solution}{}
let minimumOfThreeNumbers (a: Nat) (b: Nat) (c: Nat): Nat =
    min (min a b) c
\end{fsharpcodeshort}

\noindent In this sample solution the placeholder was replaced by a compact solution for the assignment. Student submissions may very well contain \fsharp|match|-statements,\\
\fsharp|if|-\fsharp|else|-statements or similar structures, which are also considered to be valid solutions.\\[5pt]
The given data set consists of:
\begin{itemize}[nosep]
    \item All project templates including the unit tests
    \item All uploaded files uniquely identified with timestamps and group/team IDs
    \item Timestamps of executed tests
    \item Timestamps of deletions or updates of files by the students
\end{itemize}

\noindent With this data I reconstructed every single submission snapshot, i.e. which files were uploaded when tests were executed.
Then I used Docker containers to collect the results of the build (Command: \texttt{dotnet build}) and test process (Command: \texttt{dotnet test}) for further analysis.
The results of that will be elaborated in the following sections.

\section{Insights and Patterns} %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%


% \todo{Highlight key patterns and findings from the data.}


For all of the failed submissions, i.e. the snapshots that were unable to build successfully, I looked at all occurring error codes.
In table \ref{tbl:{fsharpErrorCodes1}} (Place 1 to 5 of the Top 10) on page \pageref{tbl:{fsharpErrorCodes1}}
and table \ref{tbl:{fsharpErrorCodes2}} (Place 6 to 10 of the Top 10) on page \pageref{tbl:{fsharpErrorCodes2}} is the top 10 given with a percentage on how much that error code makes up in relation to all errors,
as well as a short code snippet that triggers that error code. In total \(94\) different error codes occurred, the distribution is show in figure \ref{fig:{buildResultsColumnChartTop10Errors}} on page \pageref{fig:{buildResultsColumnChartTop10Errors}}.
Now I will go briefly over the two most common errors, because they alone make up over \(50\%\) of all errors:
\begin{itemize}[nosep]
    \item \textbf{FS0001:} Type mismatch
    \item \textbf{FS0058:} Offside lines
\end{itemize}
Every third error is a type mismatch (error code: FS0001), which can occur on multiple instances, when in one way or a another type \texttt{A} is expected, but it given is a value of type \texttt{B} and it is not possible to implicitly cast type \texttt{B} to type \texttt{A}.\\
A few instances of that being:
\pagebreak
\begin{itemize}[nosep]
    \item \textbf{Assigning variables/attributes:}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: This expression was expected to have type 'int' but here has type 'string'.%
        }
        let x: int = "Hello"
        \end{fsharpcodeshort}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: This expression was expected to have type 'int' but here has type 'string'.%
        }
        type A = {X: int}
        let a: A = {X = "Hello"}
        \end{fsharpcodeshort}
        % \begin{itemize}[nosep]
        %     \item \fsharp|let x: int = "Hello"|\\
        %         \textit{error FS0001: This expression was expected to have type 'int' but here has type 'string'}
        %     \item \fsharp|type A = {X: int}| followed by \fsharp|let a: A = {X = "Hello"}|
        % \end{itemize}
    \item \textbf{Returning values:}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: This expression was expected to have type 'bool' but here has type 'int'.%
        }
        let f (): bool = 0
        \end{fsharpcodeshort}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: This expression was expected to have type 'int' but here has type 'float'.%
        }
        let f (n: float): int = n
        \end{fsharpcodeshort}
        While this would be ok \fsharp|let f (n: int): float = n|.
        % \begin{itemize}[nosep]
        %     \item \fsharp|let f (): bool = 0|
        %     \item \fsharp|let f (n: float): int = n|, while this would be ok\\
        %         \fsharp|let f (n: int): float = n|
        % \end{itemize}
    \item \textbf{Inconsistency when branching or matching:}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: All branches of an 'if' expression must return values implicitly convertible to the type of the first branch, which here is 'string'. This branch returns a value of type 'int'.%
        }
        let foo x = if x <> 0 then "Hello" else 4711
        \end{fsharpcodeshort}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: All branches of a pattern match expression must return values implicitly convertible to the type of the first branch, which here is 'string'. This branch returns a value of type 'float'.%
        }
        match 1.0 with | 1.0 -> "Hello" | _ -> 0.0
        \end{fsharpcodeshort}
        % \begin{itemize}[nosep]
        %     \item \fsharp|let foo x = if x <> 0 then "Hello" else 4711|\\
        %         The error message for this varies a little bit different: \textit{error FS0001: All branches of an 'if' expression must return values implicitly convertible to the type of the first branch, which here is 'string'. This branch returns a value of type 'int'.}
        %     \item \fsharp{match 1.0 with | 1.0 -> "Hello" | _ -> 0.0}\\
        %         Similarly to the example above: \textit{ error FS0001: All branches of a pattern match expression must return values implicitly convertible to the type of the first branch, which here is 'string'. This branch returns a value of type 'float'.}
        % \end{itemize}
    \item \textbf{Calling functions wrong:}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: This expression was expected to have type 'int' but here has type 'int * int'.%
        }
        let multiply x y = x * y
        multiply (1, 2)
        \end{fsharpcodeshort}
        Correct would be \fsharp|multiply 1 2|.
    \item \textbf{Operator precedence, i.e. the order of how operators are applied:}
        \begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
            \textcolor{red}{error FS0001}: The type 'string' does not match the type 'unit'.%
        }
        let s = printf "What is your name? "; System.Console.ReadLine ()
        printfn "Your name is: " + s
        \end{fsharpcodeshort}
        Correct would be \fsharp|printfn ("Your name is: " + s)|.
\end{itemize}
\pagebreak
Every fifth error has to do with offside lines/indentation (error code: FS0058) which means that some structured construct such as \fsharp|if|/\fsharp|then|/\fsharp|else|, \fsharp|try|/\fsharp|with|, \fsharp|let|, etc.
introduced an offside line and the indentation is not the same at any line of that block or there is no indentation to begin with.
Examples include when an implementation of a method is places on a new line after the method signature and there is no offset of the code or inconsistent offset.

\begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
    \textcolor{red}{error FS0058}: Unexpected syntax or possible incorrect indentation: this token is offside of context started at position (1:1). Try indenting this further.
    To continue using non-conforming indentation, pass the '--strict-indentation-' flag to the compiler, or set the language version to F\# 7.%
}
let foo (x: int) =
if x <> 0 then "A" else "B"
\end{fsharpcodeshort}

\begin{fsharpcodeshort}{}{\ttfamily\raggedright\footnotesize
    \textcolor{red}{error FS0058}: The '|' tokens separating rules of this pattern match are misaligned by one column. Consider realigning your code or using further indentation.%
}
match 1.0 with
 | 1.0 -> "A"
| _ -> "B"
\end{fsharpcodeshort}
\noindent To illustrate an actual example of that is the method to compute the minimum of three numbers \texttt{minimumOfThreeNumbers}:

\begin{fsharpcodeshort}{}{}
let minimumOfThreeNumbers (a: Nat) (b: Nat) (c: Nat): Nat = 
    failwith "TODO"
\end{fsharpcodeshort}
\noindent Here is an example of what a random submission for that assignment looks like:

\begin{fsharpcodeshortwithspaces}{}{\ttfamily\raggedright\footnotesize
    \textcolor{red}{error FS0058}: Unexpected syntax or possible incorrect indentation: this token is offside of context started at position (7:1). Try indenting this further.To continue using non-conforming indentation, pass the '--strict-indentation-' flag to the compiler, or set the language version to F\# 7.%
}
let minimumOfThreeNumbers (a: Nat) (b: Nat) (c: Nat): Nat = 
if a <= b then
     if a <= c then a else c
else
    if b <= c then b else c
\end{fsharpcodeshortwithspaces}
\noindent which is algorithmically not wrong, in fact indenting the code correctly like
\begin{fsharpcodeshortwithspaces}{}{}
let minimumOfThreeNumbers (a: Nat) (b: Nat) (c: Nat): Nat = 
    if a <= b then
        if a <= c then a else c
    else
        if b <= c then b else c
\end{fsharpcodeshortwithspaces}
\noindent will not only compile but implement the method to have the desired purpose.
