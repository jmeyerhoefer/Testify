\subsection{Value extraction for unit tests and property-based tests} %+++++++++
\label{ssctn:value-extraction-for-unit-tests-and-property-based-tests} %++++++++
%===============================================================================

While quotations preserve the structure of a test claim, Assertify still needs
to \emph{separate} the claim into the parts that correspond to the output model
(i.e., tested expression, expected value, and actual value) and then
\emph{evaluate} those parts. This is achieved by restricting the supported
shape of claims to a normal form that is both common in introductory exercises
and directly compatible with the reporting scheme: an equality-shaped claim of
the form $\textit{actual} = \textit{expected}$.

\skipM In practice, such claims are represented as quoted boolean expressions.
At runtime, Assertify and Checkify inspect these quotations using standard
quotation patterns. In the intended case, the quotation contains a call to the
equality operator, which can be deconstructed into its left-hand side (the
computed result) and right-hand side (the specification result). This
deconstruction step forms the bridge from quoted code to the three-field output
model: the \emph{tested expression} is rendered from the left-hand side (with
concrete inputs substituted, if necessary), while \emph{actual} and
\emph{expected} are obtained by evaluating the left and right subexpressions
respectively.

%===========================================================
\paragraph{Unit tests.}
\label{par:value-extraction-unit-tests}
For ordinary unit tests, the quoted claim already contains concrete values.
Accordingly, extraction proceeds directly by deconstructing the equality-shaped
quotation into its two subexpressions and evaluating them independently.
Conceptually, this maps
\begin{align*}
\textit{actual expr (lhs)}\ &\xrightarrow[]{\textit{eval}}\ \textit{actual
value}\\
&\textit{and}\\
\textit{expected expr (rhs)}\ &\xrightarrow[]{\textit{eval}}\ \textit{expected
value.}
\end{align*}

\skipM This also ensures that no solution-relevant information is leaked to
students in raw syntactic form. For example, in a claim such as
\fsharp|add 1 2 = 1 + 2|, the critical right-hand side is not exposed as source
code in the feedback, but only through its evaluated result. This supports the
didactic goal of revealing enough information to diagnose the error while
avoiding direct disclosure of the reference solution.

%===========================================================
\paragraph{Property-based tests.}
\label{par:value-extraction-property-tests}
For property-based tests, the same deconstruction principle applies, but an
additional instantiation step is required. A property is typically written as a
quoted lambda expression, for example
\fsharp|<@ fun (a:int) (b:int) -> add a b = a + b @>|, and therefore does not
initially contain the concrete inputs that caused the failure. Instead, these
inputs are generated by \textsc{FsCheck} during test execution.

\skipM To make the failing case reportable in the same format as a unit test,
Checkify executes \textsc{FsCheck} with a custom runner. When a counterexample
is found, the runner records the concrete input parameters that led to the
failure. These recorded values are then programmatically applied to the quoted
lambda expression, thereby instantiating the abstract property with the failing
inputs. For example, if the generated inputs are \fsharp|1| and \fsharp|2|, the
property
\fsharp|<@ fun (a:int) (b:int) -> add a b = a + b @>|
is transformed into the concrete quoted claim
\fsharp|<@ add 1 2 = 1 + 2 @>|.

\skipM Once this instantiation has been performed, the subsequent processing is
identical to that of unit tests. The instantiated quotation is deconstructed
into left-hand side and right-hand side, the left-hand side is rendered as the
tested expression, and both sides are evaluated independently to obtain
\emph{actual} and \emph{expected}. In this way, a failing property-based test
is reduced to the same reporting format as a failing unit test, with the
additional benefit that the feedback is tied to a concrete counterexample
produced by \textsc{FsCheck}.

\skipM A practical complication is that evaluation may itself fail. This occurs,
for example, when the student implementation raises an exception, when a branch
still contains a placeholder such as \fsharp|failwith "TODO"|, or when a quoted
expression cannot be evaluated because required runtime context is unavailable.
For novice-oriented feedback, propagating such raw evaluation exceptions would
reintroduce the problem of unstructured and potentially confusing output.
Assertify therefore treats evaluation failure as a first-class case and
produces a controlled placeholder such as \enquote{non-evaluatable} instead of
using the exception as the primary feedback message. The detailed technical
handling of these fallback cases is discussed later together with the concrete
helper functions.

\skipM Overall, this mechanism provides a stable basis for reporting because it
operates on the structured representation of the asserted claim rather than on
ad-hoc parsing of framework-generated error strings. If a quotation does not
match the expected equality pattern, the mechanism cannot reliably derive
\emph{expected} and \emph{actual} values at the current stage of the
implementation. In such cases, Assertify falls back to reporting the captured
expression together with a generic failure message. This limitation is
deliberate: support for additional assertion forms is possible, but would
require separate deconstruction rules for each class of assertion, such as
inequalities, exception checks, or arbitrary boolean predicates.

\subsection{Not improved yet}

Assertify captures the tested claim using F\# quotations. Conceptually, a
quotation turns a fragment of code into a first-class value that represents the
program structure (similar to an abstract syntax tree) rather than executing it
immediately. This enables the library to retain the student-relevant part of a
test---the asserted claim---as data. For example-based tests, a claim can be
captured directly as \verb|<@ add 1 2 = 3 @>|, allowing the feedback system to
later render a harness-independent textual representation for the \emph{tested
expression} field.

For property-based tests, the same idea is applied to predicates that depend on
generated inputs. Instead of quoting a closed expression, the test captures a
quoted lambda (e.g., \verb|<@ fun a b -> actual = expected @>|), which delays
evaluation until concrete inputs are available.

\subsubsection{Value extraction for example-based tests}
\label{sec:value-extraction-example}
After capturing a quotation, the feedback mechanism must compute the
\emph{expected} and \emph{actual} values that will be shown to the student. For
equality-shaped claims, this typically means evaluating the left-hand side
(actual) and right-hand side (expected) subexpressions separately, and reporting
their results under stable labels (\emph{Expected} and \emph{Actual}). This
turns a generic ``assertion failed'' outcome into a structured explanation that
matches how novices often reason about correctness: identify what was computed
and compare it to the intended result.

\subsubsection{Why property-based tests require additional context}
\label{sec:value-extraction-property}
Property-based testing introduces an additional requirement: a failure is only
meaningful if the concrete counterexample inputs are available at the point
where feedback is generated. Unlike example-based tests, the relevant inputs are
not written explicitly in the test case; they are produced by a generator and
may be shrunk by the runner. As a result, the feedback system must retain (or
retrieve) the final failing assignment of input variables and make it accessible
when evaluating the captured quotation.

This is why expression capture alone is insufficient for property-based tests.
The quotation is typically a lambda, and value extraction requires applying that
lambda to the specific counterexample values.

\subsubsection{Runner-assisted extraction for FsCheck-based tests}
\label{sec:runner-assisted-extraction}
To support this workflow, the property-based test runner must expose the
generated (and potentially shrunk) inputs of a failing case. Checkify therefore
relies on runner-level integration: when FsCheck reports a counterexample, the
runner provides the concrete input values to the reporting layer. The captured
quotation (a lambda) can then be applied to these values to evaluate the
relevant subexpressions and populate the feedback fields (\emph{tested
expression}, \emph{expected}, \emph{actual}) for the specific failing
assignment.

In other words, the runner supplies \emph{what inputs caused the failure}, and
the quotation supplies \emph{what claim was tested}; only their combination
enables a complete and reproducible failure explanation.

\subsubsection{Reduction traces: diagnostic benefit vs. oracle leakage}
\label{sec:reduction-traces}
Some quotation-based assertion tools also present a reduction trace, listing
intermediate rewrites of the asserted expression. For example, a failing
equality may be displayed together with progressively simplified subexpressions:

\begin{lstlisting}[caption={Example of a reduction trace (adapted from the Unquote user guide)},label={lst:unquote-trace}]
Test failed:

(1 + 2) / 3 = 2 3 / 3 = 2 1 = 2 false
\end{lstlisting}

Such traces can reduce cognitive load by connecting the final mismatch to
smaller, readable steps. At the same time, intermediate reductions may reveal
how expected values are computed (and thus provide unintended guidance). For
novice-oriented feedback, this creates a trade-off between diagnostic detail and
preserving the pedagogical intent of the exercise. Assertify therefore treats
reduction detail as a controllable aspect of feedback, rather than an
unconditional default.

\subsubsection{Summary}
Expression capture and value extraction are the technical prerequisites for
producing structured feedback in the form of \emph{tested expression},
\emph{expected value}, and \emph{actual value}. F\# quotations enable the
capture of the asserted claim as data; evaluation of quoted subexpressions
yields the runtime values needed for reporting. For property-based testing, an
additional mechanism is required: the runner must retain and provide
counterexample inputs so that a quoted lambda predicate can be applied and
evaluated in the context of the failing assignment.

\subsection{Expression capture via quotations} %++++++++++++++++++++++++++++++++
     \label{Expression capture via quotations} %++++++++++++++++++++++++++++++++
%===============================================================================


\begin{justify}
\fs quotations provide an abstract syntax tree (AST)-like representation of
code, allowing a test to pass not only a boolean result but also the syntactic
form of the tested expression. In Assertify, tests are formulated as quoted
boolean expressions (e.g., a quoted equality), which can be analysed to recover
both operands and a textual representation suitable for reporting. This directly
addresses the ``missing linkage'' problem identified in baseline assertion
failures, where expected/actual values are presented without an explicit tested
expression.
\end{justify}


%===============================================================================
\subsection{Deconstruction of equality tests} %+++++++++++++++++++++++++++++++++
     \label{Deconstruction of equality tests} %+++++++++++++++++++++++++++++++++
%===============================================================================


\begin{justify}
A primary target pattern for novice-oriented feedback is the equality
comparison, because it naturally yields an expected/actual structure. Assertify
analyses quoted boolean expressions to detect equality forms and separates
left-hand and right-hand sides. This enables the report to treat the right-hand
side as the \emph{expected} value and the left-hand side as the \emph{actual}
value, aligning the output with the consistent comparison scheme used throughout
the thesis.
\end{justify}


%===============================================================================
\subsection{Value extraction through evaluation and reduction} %++++++++++++++++
     \label{Value extraction through evaluation and reduction} %++++++++++++++++
%===============================================================================


\begin{justify}
To populate the expected and actual fields, Assertify evaluates the extracted
sub-expressions. In the simple case, both sides can be evaluated directly; in
more complex cases, evaluation may fail due to partial implementations or
non-evaluatable expressions. In such scenarios, the mechanism falls back to
reduced/decompiled representations to preserve a meaningful description rather
than producing an empty output. This ensures that reports remain informative
even when full evaluation is not possible, and it avoids silently omitting key
information.
\end{justify}

\subsection{Property-based testing and input preservation} %++++++++++++++++++++
     \label{Property-based testing and input preservation} %++++++++++++++++++++
%===============================================================================


\begin{justify}
For property-based tests, value extraction must account for generated inputs,
especially when FsCheck shrinks counterexamples. The project notes describe a
dedicated runner-based mechanism that captures the original and the shrunk
inputs for the failing test case. These captured values are then used to (i)
report the counterexample explicitly and (ii) reconstruct a readable tested
expression instantiated with the shrunk input, so that students can immediately
reproduce and reason about the minimal failing case.
\end{justify}
